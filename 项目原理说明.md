# Electrochemical Condition Agent - 项目原理说明

## 一、项目概述

**项目名称**：电化学反应条件优化智能体（Condition Agent）

**目标**：帮助研究人员为电催化烯烃双官能化反应自动设计最优反应条件。

**核心思路**：将 **LLM 推理（Claude ReAct）**、**预训练机器学习模型**、**反应相似性数据库（RAG）** 和 **反应结构可视化（RDKit Draw）** 结合，实现数据驱动 + 化学知识推理的条件推荐系统。支持多模型辩论模式（Claude + GPT + Gemini 协作）提高推荐可靠性。

**数据规模**：
- 有产率反应：`alkene_reactions_by_yield/` — 10,635 条电化学烯烃反应（11 个 JSON 文件）
- 失败反应：`extracted_smiles.json` — 327 条（yield=0）
- 合并去重后：10,635 条（有效 10,630 条）
- 训练/验证/测试：8,508 / 1,063 / 1,064（80%/10%/10%）
- FAISS 索引：10,388 条唯一「反应+条件」组合（仅从 alkene 原始数据构建，不含 train.pt 以避免重复）

**数据修正记录**：
- 2026-03-24：完整重训全部模型。数据增加 590 条 SI 表格提取反应，词汇表从截断（electrolyte 100/solvent 100/reagent 200/catalyst 100）改为完全覆盖（547/113/460/373），电极类别从 19 扩展至 23（新增 Tin, Niobium, Mercury, Pencil Graphite），使用 electrode_normalizer 统一 282+ 阳极 / 259+ 阴极名称变体
- 2026-03-20：修复 48 条溶剂 SMILES 为 `"F"` 的错误数据（F⁻ 不是合法溶剂），全部替换为 `OC(C(F)(F)F)C(F)(F)F`（HFIP，六氟异丙醇），并重新训练全部模型和重建索引

## 三、为什么需要三个小模型 + 大模型协作

### 3.1 核心问题

电化学条件推荐面临一个根本矛盾：**精度 vs 多样性 vs 可评估性**。

- 你需要一个模型告诉你每种条件**最可能**是什么 → 精度优先
- 你需要另一个模型给你**多套组合方案**，而且条件之间要兼容 → 联合生成
- 你还需要一个模型告诉你某套方案到底**好不好用** → 量化评分

没有任何单一模型能同时做好这三件事。这就是为什么要训练三个互补的小模型。

### 3.2 三个小模型各自解决什么问题

```
┌─────────────────────────────────────────────────────────────────────────┐
│                                                                         │
│   模型 1: ConditionRecommenderXTB                                      │
│   ┌────────────────────┐                                               │
│   │ 问题：每种条件最可能 │  7 个分类头独立投票                           │
│   │ 是什么？            │  → 阳极=Pt(88%), 阴极=Ni(76%), ...           │
│   │                    │                                               │
│   │ 优点：精度最高       │  Top-1: 66.5%, Top-3: 89.0%                  │
│   │ 缺点：不考虑条件间   │  可能推出化学上冲突的组合                     │
│   │       兼容性         │  （如 Pt阳极 + Br⁻电解质）                    │
│   └────────────────────┘                                               │
│                                                                         │
│   模型 2: CategoricalCVAE                                              │
│   ┌────────────────────┐                                               │
│   │ 问题：哪些条件适合   │  64D 隐变量 z 协调 7 个条件                   │
│   │ 一起使用？          │  → 方案A{Al,Pt,MeCN,...}(P=0.15)             │
│   │                    │  → 方案B{C,Pt,DCM,...}(P=0.08)               │
│   │ 优点：生成联合方案   │  同一个 z 决定所有条件，天然兼容               │
│   │ 缺点：精度较低       │  Top-1: 44.5%, Top-3: 56.0%                  │
│   └────────────────────┘                                               │
│                                                                         │
│   模型 3: YieldPredictor                                               │
│   ┌────────────────────┐                                               │
│   │ 问题：这套条件能得   │  输入完整的反应+条件 → 输出产率               │
│   │ 到多高的产率？      │  → 方案A: 78%, 方案B: 45%                    │
│   │                    │                                               │
│   │ 角色：裁判/评分器    │  R²=0.462, MAE=12.57%                        │
│   │ 用途：给候选方案排序 │  帮助 LLM 优先推荐高产率方案                  │
│   └────────────────────┘                                               │
│                                                                         │
│   三者的协作关系：                                                       │
│                                                                         │
│   模型1（高精度单条件）──┐                                               │
│                         ├→ 模型3（产率评分排序）→ 带分数的候选方案列表    │
│   模型2（联合方案生成）──┘                                               │
│                                                                         │
└─────────────────────────────────────────────────────────────────────────┘
```

**类比**：
- 模型 1 像 7 个**专科医生**，每个人在自己领域给出最佳建议，但互相不沟通
- 模型 2 像一个**全科医生**，开的处方虽然不是每项都最优，但保证药物之间不冲突
- 模型 3 像**药效评估员**，预测每个处方的实际疗效

### 3.3 大模型（Claude LLM）在每个环节的作用

大模型不是简单的"调用工具然后拼接结果"——它是**全程的指挥官、化学专家和裁判**：

| 环节 | 大模型做什么 | 为什么小模型做不到 |
|------|------------|-------------------|
| **1. 理解意图** | 解析自然语言问题，规划调用哪些工具、什么顺序 | 小模型只接受固定格式的数值输入 |
| **2. 底物分析** | 调用 RDKit 工具后，用化学知识解读"这个官能团意味着什么约束" | 小模型不理解官能团的化学含义 |
| **3. 文献检索** | 从 FAISS 返回的 5 条相似反应中归纳规律（"都用了 Pt 阳极"） | 小模型不会跨样本归纳 |
| **4. 冲突仲裁** | 模型 1 推荐 A，文献显示 B，化学常识说 C → 综合判断选哪个 | 没有任何代码逻辑做这个融合 |
| **5. 否决不合理推荐** | 系统提示词中 180 行领域知识让 LLM 能发现 ML 模型的化学错误 | 小模型不知道"Pt + Br⁻ 不兼容" |
| **6. 生成解释** | 用化学逻辑解释**为什么**选这个条件，而不是"模型输出了这个" | 小模型只输出概率，不能解释 |
| **7. 实验方案** | 将抽象的条件选择转化为具体的实验操作步骤和安全警告 | 小模型不具备实验设计能力 |

**关键设计原则**：最终推荐完全由 LLM 在自然语言层面综合推理得出，**没有任何硬编码的融合公式或投票算法**。这意味着 LLM 能根据具体情况灵活调整权重——对于常见反应更信任模型预测，对于罕见反应更依赖化学常识。

## 四、最终方案的综合决策机制

**系统没有任何代码对三个数据源做加权融合或投票。** 最终方案完全由 Claude LLM 在对话中"阅读理解"所有工具返回的结果后，结合系统提示词中的领域知识，用自然语言推理得出。

Claude 综合以下信息源做出判断：

| 信息源 | 提供的信息 | 作用 |
|---|---|---|
| **底物分析** | 反应类型、敏感基团、兼容性警告 | 确定化学约束条件 |
| **相似反应检索 (RAG)** | 真实文献中相似反应的条件和产率 | 提供经验参考 |
| **ConditionRecommenderXTB** | 每种条件的 Top-K 推荐 + 置信度 | 高精度独立推荐 |
| **CategoricalCVAE** | 多组联合条件方案 + 联合概率 | 多样化联合方案 |
| **产率预测器** | 各候选方案的预计产率 | 方案评分排序 |
| **系统提示词领域知识** | 电极/溶剂/电解质的化学规则 | 裁判矛盾结果 |

这意味着：
- Claude 能利用领域知识否决 ML 模型的不合理推荐（如卤化反应推荐了 Pt 阳极）
- 综合过程不可精确复现——同样输入可能给出略有不同的方案
- 最终方案质量高度依赖系统提示词中领域知识的质量

## 六、ReAct 智能体循环

系统采用 **ReAct（Reasoning + Acting）** 模式，由 Claude LLM 驱动决策循环：

```
用户消息 → Claude 思考 → 调用工具 → 获取结果 → Claude 再思考 → 调用下一个工具 → ... → 输出最终推荐
```

**核心类**：`ElectrochemAgent`（agent/loop.py）

**执行流程**：

1. 将用户消息加入对话历史
2. 调用 Claude API（携带系统提示词 + 10 个工具定义）
3. 解析 Claude 返回内容：
   - `text` 块 → 推理过程（思考链）
   - `tool_use` 块 → 工具调用请求
4. 若无工具调用 → 返回最终文本，结束
5. 若有工具调用 → 执行对应工具 → 将结果作为 `tool_result` 追加到对话 → 回到第 2 步
6. 达到最大轮次（15 轮）后 → 要求 Claude 总结已有发现

**推荐工作流**（系统提示词中定义）：

| 步骤 | 工具 | 目的 |
|------|------|------|
| **1. 底物分析（必须首先执行）** | `analyze_reaction_chemistry` | 识别烯烃类型、官能团、反应类别、兼容性警告 |
| 2. 解析验证 | `parse_reaction` | 验证 SMILES 格式，理解反应化学 |
| 2b. 反应可视化 | `draw_reaction` | 解析后画出反应结构图，内联显示给用户 |
| 3. 相似检索 | `search_similar_reactions` | 从数据库中找到已知相似反应，提取电流参数 |
| 3b. 相似反应可视化 | `draw_reaction` | 画出**所有**检索到的相似反应结构图 |
| 4. 条件推荐 | `recommend_conditions_default` | 获取每类条件的 top-k 独立预测 |
| 5. 多样化生成 | `generate_conditions_cvae` | 用 CVAE 生成联合条件方案 |
| 6. 产率预测 | `predict_yield` | 对候选条件组合评分排序 |
| 7. 实验方案生成 | `estimate_experimental_parameters` + `generate_experimental_protocol` | 法拉第定律计算参数 + 生成完整实验方案 |

## 七、10 个核心工具

### 7.1 analyze_reaction_chemistry — 底物化学分析（必须首先调用）

- **功能**：分析底物官能团、烯烃类型和反应类别
- **输入**：反应 SMILES
- **输出**：
  - 烯烃分析：类型（末端/内部/环内）、取代度、电子特性（缺/富电子）、共轭情况
  - 官能团检测：醇、胺、醛、酯、卤素、芳环等 24 种
  - 反应类型推断：基于原子变化自动分类（difluorination, bromoamination, carboamination 等）
  - 敏感基团警告：氧化/还原风险及建议
  - 兼容性提示：基于底物和反应类型的特殊注意事项
- **实现**：使用 RDKit 的 `Fragments`、`Descriptors` 和键分析

### 7.2 validate_smiles — 分子验证

- **功能**：验证 SMILES 字符串，提取分子性质
- **输入**：SMILES 列表
- **输出**：每个分子的有效性、标准 SMILES、分子量、分子式、环数、LogP、TPSA、氢键供/受体数
- **实现**：使用 RDKit 的 `MolFromSmiles()` 解析，`Descriptors` 计算性质

### 7.3 parse_reaction — 反应解析

- **功能**：将反应 SMILES 拆分为反应物和产物
- **输入**：反应 SMILES（格式：`reactants>>products`）
- **输出**：反应物列表、产物列表及各自分子性质
- **实现**：按 `>>` 分割，对每个组分调用 `validate_smiles`

### 7.4 recommend_conditions_default — 条件推荐（分类模型）

- **功能**：使用 ConditionRecommenderXTB 模型独立预测 7 类反应条件的 top-k
- **输入**：反应 SMILES，top_k 参数
- **输出**：7 类条件（含催化剂）各自的 top-k 预测及置信度
- **模型原理**：
  - **不是**在所有条件组合空间中搜索，而是将问题拆解为 7 个独立的多分类问题
  - 每个分类头在预定义的有限词表中选择概率最高的选项
  - 7 个分类头共享同一个 256D 隐层表示，但各自独立输出
  - 条件之间的相关性仅通过共享编码器隐式捕获
- **实现**：
  1. 计算 6144 维反应指纹（反应物FP | 产物FP | 差异FP）
  2. 计算 33 维 xTB 量子化学特征（反应物 | 产物 | 差异）
  3. 双编码器分别处理指纹和 xTB 特征，融合为 256D 表示
  4. 7 个分类头独立输出 logits
  5. 推理时屏蔽 Unknown 索引（设 logits 为 -inf）
  6. Softmax → torch.topk 选取 top-k

### 7.5 generate_conditions_cvae — 条件生成（CVAE 模型）

- **功能**：使用条件变分自编码器生成多样化的联合条件组合
- **输入**：反应 SMILES，采样数量，温度，top_k
- **输出**：
  - 每类条件的 top-k 预测（z=0 确定性推理）
  - **联合条件方案**及联合概率（多 z 采样）
- **模型原理**：
  - 也不是在全条件空间搜索，同样在有限词表上做分类
  - 核心区别：通过 64D 隐变量 z 让 7 个条件**联动**——同一个 z 确定性地映射到一组条件组合
  - 训练时将 "反应 + 真实条件组合" 的关系压缩到隐空间 z，解码器学习从 z 重建条件
  - 推理时从 N(0,I) 采样不同的 z，每次得到不同的联合方案
  - 条件间兼容性通过共享 decoder_trunk（512D）隐式建模
- **与 Tool 6.4 的区别**：ConditionRecommenderXTB 的 7 个分类头完全独立（"各自投票"），CVAE 通过 z 协调所有分类头（"先开会再各自发言"）
- **实现**：
  1. 第一阶段：z=0 确定性预测（逐条件 top-k）
  2. 第二阶段：多次 z 采样（从 64 维潜在空间采样 N 次）
     - 每次采样 z ~ N(0, I) × temperature
     - 解码为 7 类条件的联合预测
     - 计算联合对数概率：`log_prob = Σ log P(condition_i | z)`
     - 去重并按联合概率排序

### 7.6 predict_yield — 产率预测

- **功能**：预测给定条件下的反应产率（0-100%）
- **输入**：反应 SMILES + 条件集合列表（溶剂/电解质 SMILES、电极/电解池索引、电流参数）
- **输出**：每组条件的预测产率百分比
- **实现**：
  1. 构建特征向量：6144D 反应指纹 + 2048D 溶剂指纹 + 2048D 电解质指纹 + 6D 电化学参数 = 10246D
  2. 可选：+33D xTB 量子化学特征
  3. MLP 前向推理 → Sigmoid → 缩放到 0-100%

### 7.7 search_similar_reactions — 相似反应检索

- **功能**：从数据库中检索与查询反应相似的已知反应
- **输入**：反应 SMILES，top_k，相似度阈值，最低产率筛选
- **输出**：top-k 相似反应及其产率、条件、相似度得分、条件统计
- **实现**：
  1. 计算 6144D 反应指纹
  2. L2 归一化
  3. FAISS IndexFlatIP 余弦相似度检索
  4. 过滤低于阈值（默认 0.3）的结果
  5. 可选产率感知重排序：`combined_score = 0.6 × similarity + 0.4 × yield/100`
  6. 聚合条件统计（最常用的阳极/阴极/溶剂等）
  7. 提取 current_mA 和 current_density_mA_cm2 供产率预测使用

### 7.8 estimate_experimental_parameters — 实验参数计算

- **功能**：基于法拉第定律计算电化学实验参数
- **输入**：底物量(mmol)、转移电子数、电流(mA)、法拉第效率(%)、目标电流密度(mA/cm²)
- **输出**：
  - 理论电荷量 (C)
  - 实际需通过的电荷量 (C)
  - 法拉第当量 (F equiv)
  - 估计反应时间 (h)
  - 推荐电极面积 (cm²)
- **公式**：Q = n × F × z / FE（法拉第定律）

### 7.9 generate_experimental_protocol — 实验方案生成

- **功能**：生成完整的湿实验操作方案
- **输入**：阳极、阴极、电解池类型、溶剂、电解质、试剂、催化剂、底物量、电流、时间等
- **输出**：包含以下内容的结构化方案：
  - 电极准备步骤（查询 chemistry_kb.json 知识库）
  - 电解池组装指导
  - 试剂添加顺序
  - 电解参数设置
  - 后处理步骤
  - 安全提示（溶剂安全、通用安全）

### 7.10 draw_reaction — 反应结构可视化

- **功能**：使用 RDKit 绘制反应结构图，生成 base64 编码的 PNG 图片
- **输入**：
  - `reaction_smiles`（单条）或 `reaction_smiles_list`（多条批量绘制）
  - `legends`：每条反应的标签（如 "Similar #1 (yield: 85%)"）
- **输出**：base64 PNG 图片列表（`data:image/png;base64,...`）
- **实现**：
  1. `AllChem.ReactionFromSmarts(smiles, useSmiles=True)` 构建反应对象
  2. `Draw.ReactionToImage(rxn, subImgSize=(300, 200))` 渲染为 PIL Image
  3. PIL → BytesIO → base64 编码
- **Token 优化**：在 `agent/loop.py` 中，`draw_reaction` 的结果在发送给 Claude 前会剥离 `image_base64` 字段（图片数据不需要 LLM 处理），完整结果仅传给 `display_callback` 用于 Web UI 渲染
- **Web UI 集成**：
  - **Agent Chat**：tool_result 事件中检测 `draw_reaction`，将 base64 渲染为 Markdown 内联图片 `![legend](data:image/png;base64,...)`
  - **Tool Demo**：每个工具的结果下方自动附带结构图（使用 `_render_rxn_image` / `_render_mol_image` 辅助函数），无需单独的画图工具入口

## 八、机器学习模型详解

### 8.1 ConditionRecommenderXTB+Catalyst — xTB增强多头条件分类器（生产模型）

**文件**：`yield_prediction/models/condition_recommender.py`

**功能**：输入反应指纹 + xTB量子化学特征，独立预测 7 类反应条件（含催化剂）。

**网络结构**：

```
输入:
  反应指纹 (6144D) ──► FP Encoder ──► 256D
  xTB特征 (33D)    ──► xTB Encoder ──► 64D
         │                              │
         └──────────► Concat ◄──────────┘
                        │ 320D
                        ▼
┌─────────────────────────────────────────┐
│     Fusion Layer                        │
│  Linear(320 → 256)                      │
│  BatchNorm → ReLU → Dropout(0.3)        │
└──────────────┬──────────────────────────┘
               │ 256D 隐藏表示 (共享)
               ▼
┌──────────────────────────────────────────────────┐
│            7 个独立分类头                          │
│                                                  │
│  anode_head:       Linear(256 → 23)   阳极材料    │
│  cathode_head:     Linear(256 → 23)   阴极材料    │
│  cell_type_head:   Linear(256 → 6)    电解池类型  │
│  electrolyte_head: Linear(256 → 547)  电解质      │
│  solvent_head:     Linear(256 → 113)  溶剂        │
│  reagent_head:     Linear(256 → 460)  试剂        │
│  catalyst_head:    Linear(256 → 373)  催化剂      │
└──────────────────────────────────────────────────┘
```

**训练策略**：
- 优化器：AdamW (lr=0.001, weight_decay=0.0001)
- 可选产率加权损失：`loss = (Σ CE_i × yield_weight).mean()`，高产率反应权重大
- 词表构建：统计训练集中所有出现过的电解质/溶剂/试剂/催化剂（100% 覆盖，不截断）
- 电极名称归一化：使用 `electrode_normalizer.py` 将 282+ 阳极 / 259+ 阴极名称变体映射到 23 个标准类别
- 推理时屏蔽 Unknown 索引（logits 设为 -inf）

**xTB 特征详解**（33D）：
- 反应物特征 (11D): HOMO, LUMO, Gap, Dipole, IP, EA, Hardness, Electronegativity, Electrophilicity, Solvation Energy, Total Energy
- 产物特征 (11D): 同上
- 差异特征 (11D): 产物 - 反应物

**性能**：66.5% Top-1 avg（2026-03-24 基于完整词汇表+23电极类重训，100 epochs）

### 8.2 GNNConditionRecommender — 图神经网络条件推荐器

**文件**：`yield_prediction/models/gnn_condition_recommender.py`

**功能**：从分子 SMILES 动态构建图，使用图神经网络学习分子表示。

**网络结构**：

```
输入:
  反应物 SMILES ──► MolecularGNN ──► 反应物图表示 (128D)
  产物 SMILES   ──► MolecularGNN ──► 产物图表示 (128D)
  xTB特征 (11D)                            │
         │                                 │
         └─────────► ReactionTransformer ◄─┘
                           │
                           ▼
                    反应表示 (128D)
                           │
                           ▼
               ┌───────────────────────┐
               │  7 个分类头            │
               │  (同 ConditionRec)     │
               └───────────────────────┘

MolecularGNN 结构:
  Node Features (42D) ──► EdgeGatedMessagePassing ×3 ──► Global Mean Pool ──► 128D

EdgeGatedMessagePassing:
  边门控机制捕捉键的重要性
  支持原子特征 (42D) 和键特征 (10D)
```

**原子特征 (42D)**：
- 原子类型 one-hot (10D)
- 杂化态 one-hot (6D)
- 形式电荷 (1D)
- 芳香性 (1D)
- 环属性 (3D)
- 连接数 (5D)
- 氢原子数 (5D)
- 手性 (3D)
- 其他属性 (8D)

**性能**：70.1% Top-1, 90.1% Top-3

### 8.3 CategoricalCVAE+Catalyst — 条件变分自编码器

**文件**：`yield_prediction/models/condition_cvae.py`

**功能**：通过变分自编码器建模条件之间的联合分布，生成多样化的条件组合。

**这是本项目的核心创新点**：CVAE 能捕捉条件之间的相互依赖关系（例如：特定电极材料与特定电解质更匹配），而非像 ConditionRecommender 那样独立预测。

**网络结构**：

```
训练阶段:

反应指纹+xTB (6177D)──► Reaction Encoder ──► 反应表示 (256D)─┐
                        (6177→512→256)                       │
                                                            ├─► VAE Encoder ──► μ, σ ──► z (64D)
条件标签 (7个) ──► Condition Embeddings ──► 条件表示 (168D)┘    (424→512→256→64)
                   anode_emb(23,16)                                     │
                   cathode_emb(23,16)                                   │
                   cell_type_emb(6,8)                                   │
                   electrolyte_emb(547,32)              反应表示 (256D)──┤
                   solvent_emb(113,32)                                  │
                   reagent_emb(460,32)                                  ▼
                   catalyst_emb(373,32)                 VAE Decoder (320→512→512)
                                                              │
                                                              ▼
                                                       7 个分类头 → 条件预测

推理阶段:
  确定性模式: z = 0 → 解码 → 单一最优预测
  采样模式:   z ~ N(0,I) × N次 → 解码 → 多样化方案 → 按联合概率排序
```

**训练损失**：`L = Σ CE_i + β × KL`
- 重建损失：7 个条件的交叉熵之和
- KL 散度：让隐空间接近 N(0,I)
- β 预热：前 20 个 epoch 线性增长（0 → 0.1），防止 KL 坍缩

**关键维度**：
- 反应表示：256D
- 潜在空间：64D
- 条件嵌入总维度：168D（16+16+8+32+32+32+32）
- VAE 编码器输入：424D（256+168）
- VAE 解码器输入：320D（256+64）

**性能**：65.9% Top-1, 87.7% Top-3（2026-03-20 基于修正数据重训，100 epochs，best at epoch 92）

### 8.4 YieldPredictor — 产率预测器

**文件**：`yield_prediction/models/yield_predictor.py`

**功能**：给定反应和具体条件，回归预测产率（0-100%）。

**网络结构**：

```
输入特征拼接 (10246D 或 10279D):
  ┌──────────────────────────────────────────────────┐
  │ 反应物指纹 (2048D)                                │
  │ 产物指纹   (2048D)                                │
  │ 差异指纹   (2048D)  = |产物FP - 反应物FP|         │
  │ 溶剂指纹   (2048D)                                │
  │ 电解质指纹 (2048D)                                │
  │ 电化学参数 (6D): 阳极idx, 阴极idx, 电流,          │
  │                  电流密度, 电位, 电解池idx         │
  │ [可选] xTB特征 (33D)                              │
  └──────────────────┬───────────────────────────────┘
                     │
                     ▼
  ┌──────────────────────────────────────┐
  │  MLP Encoder                        │
  │  Linear(10246 → 512)                │
  │  BatchNorm → ReLU → Dropout(0.2)    │
  │  Linear(512 → 256)                  │
  │  BatchNorm → ReLU → Dropout(0.2)    │
  │  Linear(256 → 128)                  │
  │  BatchNorm → ReLU → Dropout(0.2)    │
  └──────────────┬───────────────────────┘
                 │
                 ▼
  ┌──────────────────────────────────────┐
  │  Output Layer                       │
  │  Linear(128 → 1) → Sigmoid → ×100  │
  │  输出: 预测产率 (0-100%)             │
  └──────────────────────────────────────┘
```

**关键改进**：引入 327 条失败反应（yield=0），使用 `WeightedRandomSampler` 以 2x 权重采样失败样本。

**性能**：R²=0.462, MAE=12.57%, RMSE=17.06%（2026-03-24 完整数据重训，best at epoch 17）

### 8.5 MultiTaskYieldModel — 多任务失败检测模型

**文件**：`yield_prediction/scripts/train_contrastive_v2.py`

**功能**：同时预测产率（回归）和反应是否失败（二分类），使用 Focal Loss 处理类别不平衡。

**特点**：
- 使用加权随机采样平衡失败/成功样本

## 九、系统提示词中的电化学领域知识

系统提示词（`agent/prompts.py` SYSTEM_PROMPT）中内嵌了约 180 行领域知识，作为 Claude 的"化学常识底线"，在 ML 模型推荐与化学原理矛盾时起裁判作用。

### 9.1 电极材料化学

**阳极（6 类详细描述）**：

| 材料 | 关键知识 |
|---|---|
| Graphite/Carbon rod | 便宜、惰性、宽氧化窗口 (~+2V vs SCE)，标准有机氧化选择 |
| Platinum (Pt) | O₂析出过电位高，选择性氧化优异，贵但可复用 |
| Boron-doped Diamond (BDD) | 极端氧化窗口 (>+3V vs SCE)，产生·OH 自由基 |
| RVC | 高比表面积（多孔 3D），适合低浓度底物或流动池 |
| Nickel foam/plate | 中等氧化窗口，酸性条件下会腐蚀 |
| 牺牲阳极 (Mg, Zn, Al) | 溶解提供金属阳离子，用于还原偶联机理 |

**阴极（5 类详细描述）**：Platinum、Nickel foam、Stainless Steel、Carbon/Graphite、Zinc plate

**4 条兼容性硬规则**：
- 分隔池：阳极/阴极产物不兼容时必须使用
- 非分隔池：只有一个电极反应是目标时可用
- 禁止：牺牲阳极 (Mg, Zn) + 分隔池
- 卤化反应：石墨/碳阳极优选，避免 Pt

### 9.2 电解质与溶剂选择

**5 类常用电解质**：nBu₄NBF₄/NPF₆/NClO₄、LiClO₄/LiBF₄、nBu₄NI/NBr/TBAB、NaBr/KBr/KI、Et₃N·3HF

**7 种溶剂的电化学窗口**：

| 溶剂 | 窗口 (vs NHE) | 特点 |
|---|---|---|
| MeCN (乙腈) | -3 ~ +3V | 最常用，非亲核性 |
| DMF | 良好 | 极性中间体溶剂化好 |
| DMSO | 中等 | 高电位下可被氧化 |
| DCM | 中等 | 低离子电导率 |
| MeOH | ~+1.5V | 亲核性，仅用于甲氧基化 |
| TFE | 好 | 非亲核，适合阳离子中间体 |
| H₂O | +1.23V/-0.83V | 窗口窄，竞争 O₂/H₂ 析出 |

**5 条不兼容性警告**：
- 亲核溶剂 + 阳离子中间体 → 非预期捕获
- 卤化物电解质 + 低氧化电位底物 → 非选择性卤化
- 含水条件 + 裸金属阳极 → 快速腐蚀
- 强酸性电解质 + Ni 阴极 → 溶解镍
- DCM + 强还原电位 → C-Cl 键断裂

### 9.3 电流密度选择指导

| 范围 | 适用场景 |
|---|---|
| 低 (1-5 mA/cm²) | 高选择性，敏感底物 |
| 中 (5-15 mA/cm²) | 标准范围（0.1-1 mmol 规模） |
| 高 (15-50 mA/cm²) | 制备规模，需冷却/流动池 |

恒流 (galvanostatic) vs 恒压 (potentiostatic) 的适用场景说明。

### 9.4 反应类型专项知识

| 反应类型 | 知识要点 |
|---|---|
| 自由基双官能化 | 阳极引发，碳阳极，中等电流密度 |
| 卤代双官能化 | 卤化物电解质，石墨阳极优选 |
| 氨基羟基化/双胺化 | 常需分隔池，Pt 阴极 |
| 碳胺化 | 非分隔池，碳阳极，Ni 阴极 |
| 双氟化 | Et₃N·3HF，安全关键型 |

### 9.5 行为指导规则

- 必须用化学逻辑解释推理过程，不能只贴模型输出
- 解释 **为什么** 某个条件在化学上合理

- 必须以实验方案结尾
- 标注安全风险
- 对模型置信度低时诚实告知

## 十、特征表示

### 10.1 Morgan 指纹

- **类型**：Morgan 圆形指纹（等同于 ECFP4）
- **参数**：半径 = 2，位数 = 2048
- **用途**：所有分子（反应物、产物、溶剂、电解质、催化剂）的结构表征
- **多分子聚合**：对同一类多个分子的指纹取 max（OR 操作）

### 10.2 反应指纹（6144D）

```
反应指纹 = [ 反应物FP(2048D) | 产物FP(2048D) | 差异FP(2048D) ]

其中: 差异FP = |产物FP - 反应物FP|
```

差异指纹捕捉了反应前后化学空间的变化，是反应表征的关键。

### 10.3 xTB 量子化学特征（33D）

- **来源**：extended Tight Binding (xTB) 半经验量子化学计算
- **维度**：反应物 11D + 产物 11D + 差异 11D = 33D
- **特征列表**（每个分子）：
  | 特征 | 含义 |
  |------|------|
  | HOMO | 最高占据分子轨道能量 |
  | LUMO | 最低未占据分子轨道能量 |
  | Gap | HOMO-LUMO 能隙 |
  | Dipole | 偶极矩 |
  | IP | 电离势 |
  | EA | 电子亲和能 |
  | Hardness | 化学硬度 |
  | Electronegativity | 电负性 |
  | Electrophilicity | 亲电性指数 |
  | Solvation Energy | 溶剂化能 |
  | Total Energy | 总能量 |

- **重要性**：xTB 特征使模型准确率提升约 2-3%
- **缓存**：预计算存储于 `xtb_reaction_molecules.csv`（14,513 个分子），缺失分子使用零填充 + RDKit 描述符回退

## 十二、RAG — 相似反应检索

### 12.1 索引构建（build_index.py）

**数据来源**：
- **alkene_reactions_by_yield/*.json**：CAS 反应数据（实时计算指纹）
- 注：不包含 train.pt，因为 train.pt 本身就是从 alkene_reactions_by_yield 生成的，包含会导致重复

**构建流程**：
1. 从 alkene JSON 文件加载反应数据
2. 计算 6144D 反应指纹（反应物FP | 产物FP | 差异FP）
3. 去重（基于反应 SMILES + 条件组合），10,635 条 → 10,388 条唯一组合
4. L2 归一化（使内积 = 余弦相似度）
5. 构建 FAISS IndexFlatIP(6144) 索引
6. 保存索引文件（faiss_index.bin）和反应元数据（reaction_db.pkl）

### 12.2 检索过程（rag/retriever.py）

```
查询反应 SMILES
     │
     ▼
计算 6144D 反应指纹 → L2 归一化
     │
     ▼
FAISS 内积搜索（= 余弦相似度）
     │
     ▼
过滤 similarity < threshold（默认 0.3）
     │
     ▼
[可选] 产率感知重排序:
  combined_score = 0.6 × similarity + 0.4 × yield/100
     │
     ▼
返回 top-k 相似反应 + 产率 + 条件 + 条件统计
     │
     ▼
提取 current_mA / current_density_mA_cm2 供产率预测使用
```

## 十三、模型管理（惰性加载）

**类**：`ModelManager`（models/loader.py）

所有模型采用**惰性加载**策略，即在首次调用对应工具时才加载模型权重，避免启动延迟。

```python
# 首次访问 model_manager.condition_model 时才加载
# 首次访问 model_manager.cvae_model 时才加载
# 首次访问 model_manager.yield_model 时才加载
```

加载时自动从 checkpoint 的 state_dict 推断模型维度，无需手动配置。支持自动检测新旧模型架构（ConditionRecommenderXTB vs ConditionRecommender）。

**指纹缓存**：`ToolExecutor` 维护 `_fp_cache` 字典，避免同一反应在多次工具调用中重复计算指纹。

Round 2b: draw_reaction（查询反应）
  → 绘制查询反应的结构图，内联显示给用户

Round 3: search_similar_reactions
  → 从 10,388 条数据库中找到 5 条相似反应
  → 提供实验参考条件、产率和电流参数

Round 3b: draw_reaction（所有相似反应）
  → 批量绘制所有检索到的相似反应结构图

Round 4: recommend_conditions_default
  → ConditionRecommenderXTB 输出 7 类条件各自的 top-3 预测

Round 5: generate_conditions_cvae

Round 6: predict_yield
  → 对排名前 3 的条件方案预测产率

## 十七b、辩论模式（Debate Mode）

**目的**：通过多模型协作辩论提高推荐质量和可靠性。

**参与角色**：

| 角色 | 默认模型 | 能力 |
|------|---------|------|
| **Recommender A** | Claude (当前配置) | 有工具调用能力，可访问 ML 模型和数据库 |
| **Recommender B** | GPT-5.4 | 纯文本推理，基于化学知识和 A 的工具结果 |
| **Judge** | Gemini 3.1 Pro | 综合评判两个方案，给出最终裁定 |

**4 阶段流程**：

```
阶段 1: 独立推荐（并行）
  ├── Claude (A): ReAct 循环调用工具 → 生成推荐方案 A
  └── GPT (B):    接收反应信息 → 生成推荐方案 B

阶段 2: 交叉批评（并行）
  ├── GPT (B):    阅读方案 A → 指出问题和改进建议
  └── Claude (A): 阅读方案 B → 指出问题和改进建议

阶段 3: 修正（并行）
  ├── Claude (A): 根据 B 的批评修正方案 A
  └── GPT (B):    根据 A 的批评修正方案 B

阶段 4: 最终裁决
  └── Gemini (Judge): 比较修正后的两个方案 → 综合评判 → 输出最终推荐
```

**实现细节**：
- `agent/debate_config.py`：配置类，存储模型名、API 设置
- `agent/debate_prompts.py`：各阶段的提示词模板
- `agent/debate_orchestrator.py`：编排器，使用 `ThreadPoolExecutor` 并行执行独立阶段
- GPT/Gemini 通过 OpenAI-compatible proxy 调用（使用 `openai` SDK）
- 各阶段均有优雅降级：某个模型调用失败不会阻断整个流程

**Web UI**：Settings 面板中提供 Debate Mode 复选框和 Recommender B / Judge 模型输入框。

**CLI**：`python main.py --debate --recommender-b-model gpt-5.4 --judge-model gemini-3.1-pro-preview`
